# 통신 개요

:::{admonition} 이 장에서 배우는 것
:class: note
- 로봇 통신을 어떤 목적별로 나눠서 이해하면 좋은지
- Dashboard / Socket / RTDE / 기타 인터페이스를 언제 고르면 좋은지
- 초보자가 통신을 시작할 때의 권장 순서
:::

## 초보자가 통신을 어려워하는 이유

통신은 보통 처음부터 포트 번호와 라이브러리 이름으로 들어가면 금방 복잡해집니다. 그래서 먼저 **어떤 문제를 해결하려는 통신인지**부터 나누는 것이 좋습니다.

## 목적별로 나누면 쉽다

| 하고 싶은 일 | 먼저 볼 것 | 설명 |
| --- | --- | --- |
| 프로그램 재생/정지/로드 | Dashboard | GUI 쪽 명령을 간단히 원격 제어 |
| 로봇 상태를 주기적으로 읽기 | RTDE | 구조화된 데이터 교환에 적합 |
| 로봇이 외부 장치와 직접 메시지 주고받기 | URScript Socket | 카메라, PC, 커스텀 서버와 연동 |
| URScript를 외부에서 보내기 | Primary / Secondary 계열 | 동적 프로그램 전송/제어 |

## 초보자 추천 학습 순서

### 1단계. Dashboard

가장 단순합니다. "play", "stop", "load" 같은 동작을 먼저 이해하기 좋습니다.

### 2단계. Socket

URScript에서 외부 PC와 주고받으며, 변수 기반 작업으로 확장하기 좋습니다.

### 3단계. RTDE

주기적으로 안정적으로 상태를 읽고 쓰는 구조를 배우기에 좋습니다.

## 시작 전에 꼭 해야 할 4가지

1. 로봇과 PC의 IP 대역 확인
2. ping 등으로 네트워크 연결 확인
3. 방화벽/보안 정책 확인
4. 원격 제어 권한 또는 운전 모드 확인

## 통신 선택 가이드

### Dashboard가 맞는 경우

- 상위 PC에서 프로그램 실행/정지
- 운영 상태 확인
- 아주 단순한 원격 제어

### URScript Socket이 맞는 경우

- 카메라 결과 좌표 받기
- 외부 서버에 문자열/숫자 보내기
- 커스텀 장비와 간단한 프로토콜 구성

### RTDE가 맞는 경우

- 주기적으로 TCP, I/O, 상태값 수집
- 상위 소프트웨어에서 상태 모니터링
- 실시간에 가까운 구조화 데이터 교환

## 초보자가 자주 놓치는 부분

### 좌표계와 단위
통신으로 받은 좌표가 Base 기준인지, Feature 기준인지, mm인지 m인지 확인하지 않으면 바로 문제가 생깁니다.

### 누가 서버이고 누가 클라이언트인지
Socket 예제에서는 로봇이 클라이언트인지, PC가 서버인지 역할이 자주 바뀝니다. 문서에 명확히 표기하세요.

### 프로그램과 통신을 분리하지 않음
통신 문제인지, 로봇 프로그램 문제인지 구분하기 어려워집니다. 처음에는 통신만 단독 테스트하는 편이 좋습니다.

## 사내 메뉴얼 작성 팁

통신 장에는 아래 표를 꼭 넣으세요.

| 항목 | 예시 |
| --- | --- |
| 로봇 IP | 192.168.0.10 |
| 상위 PC IP | 192.168.0.20 |
| 사용 인터페이스 | Dashboard / RTDE / Socket |
| 방향 | PC -> Robot / Robot -> PC |
| 데이터 내용 | play, 상태값, 좌표, 검사 결과 |
| 단위 | mm / m / rad / bool |

## 입문자용 원칙

- 시작은 Dashboard
- 좌표를 주고받으려면 Socket
- 상태를 안정적으로 수집하려면 RTDE
- 통신 설계 문서에 역할, 포트, 단위를 항상 같이 적기

## 체크리스트

- [ ] 목적별로 통신 방식을 고를 수 있다
- [ ] 네트워크 연결을 먼저 확인해야 한다는 점을 안다
- [ ] 역할(서버/클라이언트)과 단위를 문서에 적어야 한다는 점을 이해했다
- [ ] 다음 단계로 Dashboard, Socket, RTDE를 순서대로 학습할 준비가 됐다
