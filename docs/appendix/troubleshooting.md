# 트러블슈팅

문제가 생기면 코드를 바로 고치기보다 **증상 -> 가능한 원인 -> 첫 확인 항목** 순으로 접근하는 것이 훨씬 빠릅니다.

## 자주 만나는 증상별 정리

| 증상 | 먼저 의심할 것 | 첫 확인 항목 |
| --- | --- | --- |
| MoveL이 비틀리거나 이상하다 | TCP / Feature | active TCP, 기준 Feature, 접근 방향 |
| 같은 점인데 반복 위치가 어색하다 | Feature / Payload / 툴 체결 | 작업물 기준 좌표, payload 상태, 툴 고정 상태 |
| pick 후에만 움직임이 달라진다 | Payload / CoG | 빈 상태와 파지 후 상태를 나눴는지 |
| 베이스 근처에서 자세가 자꾸 꼬인다 | Singularity | inner workspace limit 부근인지 |
| 손목이 불필요하게 많이 돈다 | Wrist alignment | 접근 방향, MoveJ/MoveL 분리 여부 |
| Dashboard 연결은 되는데 명령 거부 | 운전 모드 / 권한 / 안전 상태 | robotmode, remote control 가능 여부 |
| Socket 연결은 되는데 좌표가 이상하다 | 단위 / 좌표계 / 문자열 형식 | mm/m, deg/rad, Base/Feature 기준 |
| RTDE 연결은 되는데 값 해석이 안 된다 | recipe / 좌표 의미 | 어떤 신호를 읽는지, active TCP가 무엇인지 |
| URSim에서는 되는데 실제 장비에서 안 된다 | 실제 하드웨어 조건 | 충돌 가능성, payload, I/O, 실제 신호 |

## 증상별 빠른 대응

### 1. 작업점이 계속 어긋난다

순서대로 보세요.

1. TCP가 맞는가
2. Move 노드 기준 Feature가 맞는가
3. 작업물이 실제로 같은 위치인가
4. 툴 체결이 변하지 않았는가

### 2. pick 후 보호정지가 난다

- payload 전환이 있는가
- 작업물 무게가 반영됐는가
- 접근/이탈 경로가 무리하지 않은가

### 3. 통신으로 받은 좌표가 이상하다

- 문자열 원문을 저장한다
- 단위를 문서와 대조한다
- 기준 좌표를 확인한다
- 받은 pose를 바로 작업점으로 쓰지 말고 접근점도 계산한다

### 4. 프로그램이 복잡해져서 디버깅이 안 된다

- GUI 흐름과 Script 계산을 분리한다
- 로그용 popup / textmsg를 잠시 넣는다
- 점 이름과 변수명을 의미 있게 바꾼다

## 문제를 빨리 푸는 습관

### 습관 1. 한 번에 하나만 바꾼다
TCP, Feature, Payload를 동시에 바꾸면 원인을 못 찾습니다.

### 습관 2. 공회전 경로를 따로 둔다
실제 작업 높이 바로 아래로 가지 말고, 항상 접근 높이에서 먼저 검증하세요.

### 습관 3. 문서에 "증상-원인-해결"을 남긴다
사내 문서는 경험 누적이 핵심입니다.

## 사내 확장 양식 예시

```text
발생일:
설비명:
프로그램명:
증상:
직전 변경사항:
원인:
조치:
재발 방지:
```

## 우선순위 규칙

문제가 생겼을 때는 보통 아래 순서로 보는 것이 효율적입니다.

1. 안전 상태 / 알람
2. TCP / Feature
3. Payload
4. 경로 설계(MoveJ/MoveL/접근점)
5. 통신 데이터 형식
6. Script 계산 로직

:::{note}
"로봇이 이상하다"는 증상은 실제로는 좌표, payload, 특이점, 통신 단위 중 하나일 가능성이 높습니다. 증상을 추상적으로 두지 말고, 어떤 점에서 어떤 방향으로 어떻게 이상한지 기록하세요.
:::
